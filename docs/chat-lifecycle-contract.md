# Chat Lifecycle Contract

This is the Part A contract for the Chats lifecycle work. It turns the current
hook-to-frontmatter behavior into an event producer contract that later tasks can
implement without changing the product model again.

Sources used:

- Existing Hitch hook scripts in `desktop/src/main/main.ts`.
- Existing daemon launch/link/settle callbacks in `daemon/src/daemon.ts`.
- OpenAI Codex hook docs: https://developers.openai.com/codex/hooks
- Claude Code hook docs: https://docs.anthropic.com/en/docs/claude-code/hooks
- Redacted fixtures in `docs/fixtures/chat-lifecycle-hooks/`.

## Hook Payload Scope

Hitch cares about lifecycle signals, not transcript content.

Codex events to capture:

| Provider event | Required fields | Optional useful fields | Normalized lifecycle | Status |
| --- | --- | --- | --- | --- |
| `UserPromptSubmit` | `hook_event_name`, `cwd`, chat id candidate | `turn_id`, `prompt` | `turn.started` | `working` |
| `PreToolUse` | `hook_event_name`, `cwd`, chat id candidate | `turn_id`, `tool_name`, `tool_use_id` | `turn.resumed` | `working` |
| `PermissionRequest` | `hook_event_name`, `cwd`, chat id candidate | `turn_id`, `tool_name` | `turn.needs_input` | `needs-input` |
| `Stop` | `hook_event_name`, `cwd`, chat id candidate | `turn_id` | `turn.completed` | `waiting` |

Codex chat id candidates, in order: `session_id`, `sessionId`, `thread_id`,
`threadId`, `thread.id`, `CODEX_THREAD_ID`, then a UUID parsed from
`transcript_path` if present. Keep this tolerant because current Hitch already
does.

Claude Code events to capture:

| Provider event | Required fields | Optional useful fields | Normalized lifecycle | Status |
| --- | --- | --- | --- | --- |
| `SessionStart` | `hook_event_name`, `session_id`, `cwd` | `source`, `model`, `session_title` | `session.started` | no status change |
| `UserPromptSubmit` | `hook_event_name`, `session_id`, `cwd` | `prompt` | `turn.started` | `working` |
| `PreToolUse` | `hook_event_name`, `session_id`, `cwd` | `tool_name` | `turn.resumed` | `working` |
| `Notification` with `notification_type=permission_prompt` | `hook_event_name`, `session_id`, `cwd`, `notification_type` | `message` | `turn.needs_input` | `needs-input` |
| `Stop` | `hook_event_name`, `session_id`, `cwd` | `background_tasks`, `session_crons` | `turn.completed` | `waiting` |
| `SessionEnd` | `hook_event_name`, `session_id`, `cwd`, `reason` | none | `session.ended` | no status change; clear process liveness |

Ignore Claude `Notification` payloads whose `notification_type` is not
`permission_prompt`.

## Normalized Event Shape

Hooks and daemon callbacks both write this event shape to SQLite:

```ts
type ChatLifecycleStatus = "working" | "needs-input" | "waiting" | "idle";

type ChatLifecycleEvent = {
  schemaVersion: 1;
  eventId: string;
  source: "hook" | "daemon";
  producer:
    | "codex-hook"
    | "claude-code-hook"
    | "daemon-launch"
    | "daemon-linker"
    | "daemon-appserver"
    | "daemon-reconcile";
  harness: "codex" | "claude-code";
  providerEvent: string;
  lifecycle:
    | "chat.created"
    | "chat.bound"
    | "turn.started"
    | "turn.resumed"
    | "turn.needs_input"
    | "turn.completed"
    | "session.started"
    | "session.ended";
  status: ChatLifecycleStatus | null;
  projectId: string | null;
  projectLocalPath: string | null;
  chatId: string | null;
  launchId: string | null;
  turnId: string | null;
  cwd: string;
  host: string;
  observedAt: string;
  rawPayloadHash: string | null;
  rawPayloadRef: string | null;
  metadata: Record<string, unknown>;
};
```

Rules:

- `chatId` is nullable only before a pending Hitch launch is bound to a real
  harness id.
- `launchId` is required for Hitch-started pending chats and should stay on the
  row after binding for traceability.
- `observedAt` is generated by the producer at insert time.
- `projectId` may be null in the hook. The reducer resolves it from `cwd` using
  the first enabled Hitch root that contains `cwd`.
- `metadata` is for non-secret operational details only: tool name,
  notification type, environment, session-start source, session-end reason,
  command id, or app-server result.

## Dedupe

Use two layers:

1. Producer idempotency with `eventId`.
2. Reducer no-op suppression when the reduced chat state does not change.

Hook `eventId`:

```ts
sha256(JSON.stringify({
  source,
  producer,
  harness,
  providerEvent,
  chatId,
  launchId,
  turnId,
  toolUseId,
  status,
  rawPayloadHash,
}))
```

Daemon `eventId`:

```ts
sha256(JSON.stringify({
  source,
  producer,
  harness,
  providerEvent,
  chatId,
  launchId,
  commandId,
  status,
  observedTransition,
}))
```

Create a unique index on `chat_events.event_id`. On duplicate insert, ignore and
still touch the bump file only if the insert succeeded. The reducer should sync
Convex only when `local_chats` changes.

## Raw Payload Retention

Default: do not persist full raw hook payloads.

Persist `rawPayloadHash`, `providerEvent`, and selected redacted metadata. Full
raw payload retention should be opt-in debug mode only, local-only, and scrubbed:

- Drop `prompt`, `last_assistant_message`, transcript content, auth material,
  environment variables, and any `tool_response`.
- For `tool_input`, keep only `tool_name` and optionally a redacted command
  summary during debug.
- Delete debug raw payload rows after 24 hours.

The normal event log retention can keep reduced, non-raw events for 7 days after
successful reduction and sync.

## Local SQLite

Package choice: use `node:sqlite` from the Node runtime, not a repo dependency.
Reason: installed global hooks must be standalone scripts and should not resolve
native modules from Hitch's workspace. This repo currently requires Node
`>=24.15.0`; this machine's `v24.14.1` can import `node:sqlite` but prints an
experimental warning, so B2 should verify the supported packaged runtime before
shipping the hook rewrite.

Database path:

```text
<appSupportDir>/chat-lifecycle.sqlite
```

Bump file path:

```text
<appSupportDir>/chat-lifecycle.bump
```

Use the same app support root as `desktop/src/main/main.ts`:

- dev: `~/Library/Application Support/Hitch Dev`
- prod: `~/Library/Application Support/Hitch`

SQLite settings:

```sql
PRAGMA journal_mode = WAL;
PRAGMA busy_timeout = 1000;
PRAGMA foreign_keys = ON;
```

Schema:

```sql
CREATE TABLE chat_events (
  seq INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id TEXT NOT NULL UNIQUE,
  schema_version INTEGER NOT NULL,
  source TEXT NOT NULL,
  producer TEXT NOT NULL,
  harness TEXT NOT NULL,
  provider_event TEXT NOT NULL,
  lifecycle TEXT NOT NULL,
  status TEXT,
  project_id TEXT,
  project_local_path TEXT,
  chat_id TEXT,
  launch_id TEXT,
  turn_id TEXT,
  cwd TEXT NOT NULL,
  host TEXT NOT NULL,
  observed_at INTEGER NOT NULL,
  raw_payload_hash TEXT,
  raw_payload_ref TEXT,
  metadata_json TEXT NOT NULL DEFAULT '{}',
  reduced_at INTEGER
);

CREATE INDEX chat_events_by_reducer ON chat_events(seq) WHERE reduced_at IS NULL;
CREATE INDEX chat_events_by_chat ON chat_events(harness, chat_id, seq);
CREATE INDEX chat_events_by_launch ON chat_events(launch_id, seq);

CREATE TABLE local_chats (
  local_key TEXT PRIMARY KEY,
  project_id TEXT,
  launch_id TEXT,
  harness TEXT NOT NULL,
  chat_id TEXT,
  pending INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL,
  title TEXT NOT NULL,
  cwd TEXT NOT NULL,
  host TEXT NOT NULL,
  environment TEXT,
  linked_type TEXT,
  linked_path TEXT,
  resume_kind TEXT NOT NULL,
  resume_payload_json TEXT NOT NULL DEFAULT '{}',
  first_observed_at INTEGER NOT NULL,
  last_event_at INTEGER NOT NULL,
  last_status_at INTEGER NOT NULL,
  ended_at INTEGER,
  pinned INTEGER NOT NULL DEFAULT 0,
  pinned_at INTEGER,
  archived_at INTEGER,
  deleted_at INTEGER,
  dirty INTEGER NOT NULL DEFAULT 1,
  last_synced_at INTEGER,
  convex_id TEXT,
  updated_at INTEGER NOT NULL
);

CREATE UNIQUE INDEX local_chats_by_chat
  ON local_chats(harness, chat_id, host)
  WHERE chat_id IS NOT NULL;

CREATE UNIQUE INDEX local_chats_by_launch
  ON local_chats(launch_id)
  WHERE launch_id IS NOT NULL;

CREATE TABLE meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

`local_key` is `chat:<harness>:<host>:<chatId>` once bound and
`launch:<launchId>` while pending.

Migrations: store `schema_version` in `meta`; run explicit numbered migrations
inside a transaction on daemon startup and hook insert. Hooks should initialize
only the minimum schema needed to insert if the daemon has not run yet.

Reducer wakeup defaults:

- Bump write: after successful event insert, update `chat-lifecycle.bump` mtime
  or write the latest event sequence as plain text.
- Watch debounce: 100 ms.
- Poll fallback: every 5 seconds.
- Startup: run reducer once before subscribing to commands.
- Retention cleanup: after successful sync, delete reduced `chat_events` older
  than 7 days; delete debug raw payloads older than 24 hours.

## Convex `chats` Schema

Add a first-class table:

```ts
chats: defineTable({
  projectId: v.id("projects"),
  launchId: v.optional(v.string()),
  harness: v.union(v.literal("claude-code"), v.literal("codex")),
  chatId: v.optional(v.string()),
  pending: v.boolean(),
  status: v.union(
    v.literal("working"),
    v.literal("needs-input"),
    v.literal("waiting"),
    v.literal("idle"),
  ),
  title: v.string(),
  cwd: v.string(),
  host: v.string(),
  environment: v.optional(
    v.union(
      v.literal("cmux"),
      v.literal("codex-app"),
      v.literal("vscode"),
      v.literal("cursor"),
      v.literal("t3code"),
    ),
  ),
  linkedType: v.optional(v.union(v.literal("task"), v.literal("note"))),
  linkedPath: v.optional(v.string()),
  resumeKind: v.union(
    v.literal("open-chat-command"),
    v.literal("external"),
  ),
  resumePayload: v.optional(v.any()),
  firstObservedAt: v.number(),
  lastEventAt: v.number(),
  lastStatusAt: v.number(),
  endedAt: v.optional(v.number()),
  pinned: v.optional(v.boolean()),
  pinnedAt: v.optional(v.number()),
  archivedAt: v.optional(v.number()),
  deletedAt: v.optional(v.number()),
  createdAt: v.number(),
  updatedAt: v.number(),
})
  .index("by_project", ["projectId"])
  .index("by_project_status", ["projectId", "status"])
  .index("by_project_updated", ["projectId", "updatedAt"])
  .index("by_project_pinned", ["projectId", "pinned", "pinnedAt"])
  .index("by_project_archived", ["projectId", "archivedAt"])
  .index("by_project_launch", ["projectId", "launchId"])
  .index("by_project_chat", ["projectId", "harness", "chatId", "host"]);
```

Use application-level uniqueness for `launchId` and `(projectId, harness,
chatId, host)` because Convex indexes are lookup paths, not uniqueness
constraints.

Display mapping: the stored status remains `waiting`; the Chats tab may render
`waiting` as `idle` to match the locked design.

## `chats.startChat`

Add a Convex mutation that creates a pending chat row and enqueues the daemon
command in one transaction.

Input:

```ts
{
  projectId: Id<"projects">;
  harness: "claude-code" | "codex";
  initialPrompt: string;
  cwd?: string;
  host?: string;
  model?: string;
  effort?: string;
  linkedType?: "task" | "note";
  linkedPath?: string;
}
```

Mutation behavior:

- Generate `launchId`.
- Create `chats` row with `pending: true`, `chatId` unset, `status: "working"`,
  `title` derived from the first prompt, and link fields when present.
- Enqueue a `commands` row with `kind: "start-chat"`, `launchId`, `harness`,
  `initialPrompt`, `cwd`, `model`, `effort`, `linkedType`, and `linkedPath`.
- Return `{ chatId: Id<"chats">, commandId: Id<"commands">, launchId }`.

Extend `commands` with:

```ts
launchId: v.optional(v.string()),
linkedType: v.optional(v.union(v.literal("task"), v.literal("note"))),
linkedPath: v.optional(v.string()),
```

For task-linked compatibility, keep accepting the existing `path` field during
migration and normalize it to `linkedType: "task"`, `linkedPath: path`.

Daemon command payload sent to launchers:

```ts
{
  commandId: string;
  launchId: string;
  projectId: string;
  projectName: string;
  harness: "claude-code" | "codex";
  environment: Environment;
  cwd: string;
  initialPrompt: string;
  model?: string;
  effort?: string;
  linkedType?: "task" | "note";
  linkedPath?: string;
}
```

Daemon lifecycle events:

- Before launch: `chat.created`, `status: "working"`, `launchId`.
- When launcher learns the real id: `chat.bound`, `chatId`, `launchId`,
  `status: "working"`.
- Codex app-server turn completion: `turn.completed`, `status: "waiting"`.
- Claude session discovery/linking: `chat.bound`, `status: "working"`.
- Process reconciliation/session end: `session.ended`, no direct status change
  in v1 unless a later compatibility task decides to clear dead linked cards.

## Verification Notes

Run:

```sh
node scripts/normalize-chat-lifecycle-fixtures.mjs
```

This validates the redacted fixtures against the normalized event shape.

Real live payload capture was not performed in this spike. Installing temporary
global hooks would mutate the user's Codex/Claude configuration, and this task
does not need that side effect to pin the implementation contract. The fixture
shapes are based on current official docs plus the fields Hitch already parses.
Remaining uncertainty: exact Codex app/native payload id field can vary, so the
Codex normalizer must keep the tolerant candidate id extraction described above.
